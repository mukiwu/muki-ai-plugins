# shipshape-skills 自動行為規則

以下規則在安裝此 plugin 後自動生效，不需要使用者手動觸發。

## 完成驗證（Iron Law）

**這是所有規則的上位原則。宣稱完成之前，必須有驗證證據。沒有跑過驗證指令，不能說「完成了」「應該可以了」「看起來沒問題」。**

| 宣稱 | 必須提供的證據 | 不算數的 |
|------|-------------|---------|
| 測試通過 | 測試指令輸出：0 failures | 「上次跑過」「應該會過」 |
| Build 成功 | build 指令：exit 0 | linter 過了就當 build 過 |
| Bug 修好了 | 重現步驟跑過：確認正常 | 「程式碼改了，應該修好了」 |
| 功能完成 | 實際操作過或測試覆蓋到 | 「邏輯看起來對」 |

**禁止的措辭**：「應該可以了」「看起來沒問題」「我覺得修好了」「理論上沒問題」。這些都代表你還沒驗證。

**回報時帶證據**：說「測試通過，42/42 pass」而非「測試通過了」。

## 功能開發

當使用者描述想要新增功能、加入新頁面、實作新需求時，主動建議執行 `/shipshape-skills:feature` 進入完整開發流程。如果任務明顯很小（改一行設定、換個字串），不需要建議。

## Bug 修復

只要這次修改涉及「修正錯誤行為」，就自動執行 `bug-fix-learning` skill 的完整流程。觸發場景包含但不限於：使用者回報 bug、review PR 後修正寫法、修復 Sentry/監控系統報的錯、發現既有邏輯有誤而修正。評估值得沉澱時，必須實際用 Edit/Write 工具寫入檔案——讀取或口頭承認都不算完成。

## 框架 Patterns

修改 React 相關檔案（`.jsx`、`.tsx`）時，自動參考 `react-patterns` skill 的最佳實踐。
修改 Vue 相關檔案（`.vue`、Composition API）時，自動參考 `vue-patterns` skill 的最佳實踐。

## 品質關卡

完成有意義的功能修改（新增功能、修改邏輯、重構元件——不包含純 CSS 調整、換字串、改設定檔等瑣碎變更）後，依序通過以下關卡。Commit 前確認所有關卡都已通過。

### 1. 測試通過

執行專案的完整測試（如 `npm test`、`bun test`、`vitest run`），確認修改沒有破壞既有功能。

如果測試有失敗：
1. 分析失敗原因——區分「測試本身需要更新」vs「程式碼有 bug」
2. 如果是測試需要更新（因為行為預期改變了），直接修復測試使其符合新行為
3. 如果是程式碼有 bug，回去修程式碼，然後重新跑測試確認通過

### 2. Cookbook 同步

檢查這次修改是否影響了 `docs/cookbook/` 中記錄的內容：

1. 根據這次修改的檔案和函式名稱，用 Grep 搜尋 `docs/cookbook/` 中是否有提到它們
2. 如果 cookbook 中有記錄但描述已經與程式碼不一致（例如 API 用法改了、參數變了、元件行為不同了），用 Edit 工具更新 cookbook 使其與新程式碼一致
3. 如果 cookbook 中沒有相關記錄，**預設不新增**。只有通過 `bug-fix-learning` skill Step 2 的三問判準（tsc/eslint/測試/deps-check 都抓不到的隱性知識）才寫入。純實作細節、型別可推導的契約、import 關係都不寫。

### 3. Bug fix learning

如果這次修改涉及修復錯誤行為，確認已執行 `bug-fix-learning` skill 的完整流程（見「Bug 修復」區塊的觸發條件）。

### 4. Code Review

使用 `code-reviewer` agent 進行審查。如果改動很小（< 5 行、純設定變更），可以跳過。

所有關卡通過後才可以 commit。

## 測試

寫完單元測試後，主動詢問是否要執行 `auto-improve-tests` 優化測試品質。
功能開發完成後，主動詢問是否需要撰寫 E2E 測試。
